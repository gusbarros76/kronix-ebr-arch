Trigger + Interceptor (híbrido)
Trigger PostgreSQL — garante cobertura total. Qualquer escrita no banco é capturada, mesmo fora do ORM.

EF Core Interceptor — enriquece o contexto. Adiciona o actor_id (quem fez), o nome do Command CQRS e metadados de negócio que o banco não tem acesso.
📦
JSONB entity-level
1 INSERT por transação — não importa quantos campos mudaram. O before_state e after_state capturam o estado completo da entidade como JSON estruturado.

Por que não field-level: N INSERTs por transação + reconstrução de estado exige aggregation query — mais complexo para auditores e inspetores.
🗄️
Armazenamento em camadas
RDS PostgreSQL — 35 dias, quente, PITR disponível.

S3 Glacier + Object Lock WORM — 5 anos, imutável por contrato de infraestrutura. Nem o admin AWS apaga. Custo ~20x menor que RDS por GB.
✅
Premissa regulatória: ANVISA RDC 658/2022, FDA 21 CFR Part 11 e GMP Annex 11 exigem que registros eletrônicos sejam protegidos contra alteração e deleção, com rastreabilidade de quem, o quê, quando e de onde. Cada decisão acima foi tomada para satisfazer esse requisito por arquitetura — não por policy que pode ser esquecida.

02 Trigger + Interceptor — Por que os dois Cada um resolve um problema diferente · Complementares, não redundantes
🔌
EF Core SaveChangesInterceptor
Camada Application · .NET C# · Contexto de negócio
O que faz: intercepta toda chamada de SaveChanges() no EF Core antes de tocar o banco. Captura quais entidades mudaram, serializa before/after como JSONB, injeta actor_id do token SSO.
Por que é necessário: o banco não sabe quem é o usuário SSO. O actor_id — informação mais crítica do audit GxP — só existe na camada de aplicação, no Bearer Token da requisição.
O que também captura: nome do Command CQRS (SignStepCommand), contexto de negócio, correlation_id para rastreabilidade com Datadog.
Limitação conhecida: só vê o que passa pelo DbContext. Scripts diretos no banco, migrations manuais e jobs que usam ADO.NET puro não são cobertos.
ℹ️
Boa prática: o interceptor deve ser registrado uma única vez no container DI como singleton. Nunca instanciar por requisição.
⚡
PostgreSQL Trigger
Banco de dados · PL/pgSQL · Safety net total
O que faz: dispara automaticamente no banco após qualquer INSERT, UPDATE ou DELETE nas tabelas de ebr_core — independente de quem ou o quê gerou a escrita.
Por que é necessário: garante que nenhuma escrita escapa do audit trail. Migration manual, script do DBA, job de background fora do ORM — tudo é capturado. É o safety net da arquitetura.
Como lida com o actor_id: lê da variável de sessão PostgreSQL (current_setting('app.current_actor')). A aplicação seta essa variável via SET LOCAL antes de qualquer write. Se não setar, grava SYSTEM — identificando operação fora do fluxo normal.
Operações fora do ORM identificadas: actor_id = SYSTEM no registro é um alerta de investigação — algo escreveu no banco sem passar pela aplicação.
⚠️
Regra de processo: qualquer acesso direto ao banco em produção requer aprovação formal da área de Qualidade. O trigger registra — o processo garante que não acontece sem controle.

Implementação — PostgreSQL Trigger (PL/pgSQL)
ebr_audit_trigger.sql 
-- Função chamada pelo trigger em cada tabela do ebr_core
CREATE OR REPLACE FUNCTION ebr_core.audit_trigger_fn()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO ebr_audit.audit_events (
    event_id, timestamp_utc, actor_id,
    entity_type, entity_id, action,
    before_state, after_state, source
  ) VALUES (
    gen_random_uuid(),
    now() AT TIME ZONE 'UTC',
    -- Lê actor_id da sessão. Se não setado, grava SYSTEM (alerta)
    COALESCE(current_setting('app.current_actor', true), 'SYSTEM'),
    TG_TABLE_NAME,
    COALESCE(NEW.id::text, OLD.id::text),
    TG_OP,             -- INSERT | UPDATE | DELETE
    CASE TG_OP WHEN 'DELETE' THEN to_jsonb(OLD) ELSE to_jsonb(OLD) END,
    CASE TG_OP WHEN 'DELETE' THEN NULL  ELSE to_jsonb(NEW) END,
    'TRIGGER'          -- diferencia de registros vindos do Interceptor
  );
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Aplicar em cada tabela crítica do ebr_core
CREATE TRIGGER audit_batch_records
  AFTER INSERT OR UPDATE OR DELETE ON ebr_core.batch_records
  FOR EACH ROW EXECUTE FUNCTION ebr_core.audit_trigger_fn();

Implementação — EF Core Interceptor (C#)
AuditInterceptor.cs · Infrastructure/Audit/ 
public class AuditInterceptor : SaveChangesInterceptor
{
    private readonly IHttpContextAccessor _http;

    public AuditInterceptor(IHttpContextAccessor http) => _http = http;

    public override async ValueTask<InterceptionResult<int>>
        SavingChangesAsync(DbContextEventData data, ...)
    {
        var actorId = _http.HttpContext?
            .User.FindFirst("sub")?.Value ?? "anonymous";

        // Injeta na sessão PostgreSQL — o Trigger lê daqui se precisar
        await data.Context.Database
            .ExecuteSqlRawAsync($"SET LOCAL app.current_actor = '{actorId}'");

        var entries = data.Context.ChangeTracker.Entries()
            .Where(e => e.State is Added or Modified or Deleted);

        foreach (var entry in entries)
        {
            var evt = new AuditEvent {
                EventId     = Guid.NewGuid(),
                TimestampUtc = DateTime.UtcNow,
                ActorId     = actorId,
                EntityType  = entry.Entity.GetType().Name,
                EntityId    = entry.Property("Id").CurrentValue?.ToString(),
                Action      = entry.State.ToString(),
                BeforeState = entry.State == EntityState.Added
                                ? null
                                : JsonSerializer.Serialize(entry.OriginalValues.ToObject()),
                AfterState  = entry.State == EntityState.Deleted
                                ? null
                                : JsonSerializer.Serialize(entry.CurrentValues.ToObject()),
                Source      = "INTERCEPTOR"  // diferencia do Trigger
            };
            data.Context.Set<AuditEvent>().Add(evt);
        }
        return await base.SavingChangesAsync(data, ...);
    }
}
03 JSONB — Por que entity-level e não field-level Decisão fechada · 1 INSERT por transação · Estado completo em uma linha
✗ Field-level (legado SAP MII — FDB_AUDIT) Descartado
Uma linha por campo alterado. Transação que muda 8 campos = 8 INSERTs separados.
FDB_AUDIT — 8 linhas para 1 transação
-- Operador altera status + quantidade + responsável
-- Resultado: 3 linhas para 1 ação
TRAIL_ID | TABELA      | CAMPO      | VALOR_ANTES | VALOR_DEPOIS
---------|-------------|------------|-------------|-------------
  1001   | FDB_FASE    | CD_STATU   | 'EXEC'      | 'SIGN'
  1002   | FDB_FASE    | QT_REAL    | '100'       | '98.5'
  1003   | FDB_FASE    | CD_USUAR   | NULL        | 'user_x'

-- Para reconstruir o estado: query de agregação complexa
SELECT CAMPO, VALOR_ANTES, VALOR_DEPOIS
FROM FDB_AUDIT
WHERE ROW_ID = '123' AND DATAHORA = '...'
ORDER BY TRAIL_ID; -- auditor monta na cabeça
Problemas: write amplification · reconstrução de estado exige aggregation · configuração manual de quais campos auditar (FDB_AUDIT_CONDICOES) — risco de campo esquecido
✓ Entity-level JSONB (Kronix EBR) Escolhido
Uma linha por transação — independente de quantos campos mudaram. Estado completo antes e depois.
audit_events — 1 linha para a mesma transação
-- Mesma operação: 1 único INSERT
event_id  | actor_id | entity_type | action | before_state          | after_state
----------|----------|-------------|--------|-----------------------|--------------------
 uuid-x   | user_x   | RecipeStep  | UPDATE | {"status":"EXEC",     | {"status":"SIGN",
          |          |             |        |  "qty":100,           |  "qty":98.5,
          |          |             |        |  "signed_by":null}    |  "signed_by":"x"}

-- Para reconstruir o estado: leitura direta
SELECT before_state, after_state
FROM ebr_audit.audit_events
WHERE entity_id = 'step-uuid'
ORDER BY timestamp_utc; -- auditor lê direto
Benefícios: 1 INSERT por transação · snapshot completo legível · cobertura total sem configuração · PostgreSQL suporta nativamente · indexável com GIN

Schema — audit_events (ebr_audit)
schema: ebr_audit · tabela: audit_events · permissão: INSERT only 
CREATE TABLE ebr_audit.audit_events (
  event_id      UUID          DEFAULT gen_random_uuid() PRIMARY KEY,
  timestamp_utc TIMESTAMPTZ   NOT NULL DEFAULT now() AT TIME ZONE 'UTC',
  actor_id      VARCHAR(255)  NOT NULL,   -- user_id do SSO ou SYSTEM
  entity_type   VARCHAR(100)  NOT NULL,   -- BatchRecord, RecipeStep, etc.
  entity_id     VARCHAR(255)  NOT NULL,   -- UUID da entidade afetada
  action        VARCHAR(50)   NOT NULL,   -- INSERT | UPDATE | DELETE | SIGN | APPROVE
  before_state  JSONB,                     -- NULL em criações
  after_state   JSONB,                     -- NULL em deleções
  reason        TEXT,                      -- motivo obrigatório para ações GxP críticas
  source        VARCHAR(20)   NOT NULL    -- INTERCEPTOR | TRIGGER
    CHECK (source IN ('INTERCEPTOR', 'TRIGGER'))
);

-- Índices principais
CREATE INDEX idx_audit_entity     ON ebr_audit.audit_events (entity_id, timestamp_utc);
CREATE INDEX idx_audit_actor       ON ebr_audit.audit_events (actor_id, timestamp_utc);
CREATE INDEX idx_audit_entity_type ON ebr_audit.audit_events (entity_type, timestamp_utc);
CREATE INDEX idx_audit_after_gin   ON ebr_audit.audit_events USING GIN (after_state);

-- Imutabilidade: remove permissão de UPDATE e DELETE da role da aplicação
REVOKE UPDATE, DELETE ON ebr_audit.audit_events FROM ebr_app_role;
GRANT INSERT, SELECT                ON ebr_audit.audit_events TO ebr_app_role;
04 Armazenamento em Camadas — Hot · Warm · Cold Custo x Acesso x Compliance · Cada tier tem responsabilidade única
🔥 HOT
RDS PostgreSQL · schema ebr_audit
Retenção 35 dias · PITR disponível · Multi-AZ · Backup automático diário
Acesso operacional imediato PITR — restauração por segundo Queries diretas via API read-only
↓ Export diário automático (Lambda ou pg_dump agendado)
❄️ COLD
S3 Glacier + Object Lock WORM
Retenção 5 anos · Imutável — REVOKE de deleção no nível AWS · Custo ~20x menor que RDS/GB · SSE-KMS
Object Lock — nem root AWS apaga WORM — Write Once Read Many Criptografia SSE-KMS · chave dedicada EBR
✅
Por que 35 dias no RDS e não mais? RDS Multi-AZ é o armazenamento mais caro da stack — ~$0.115/GB/mês (gp3). S3 Glacier custa ~$0.004/GB/mês. Manter 5 anos no RDS seria 28x mais caro sem benefício operacional. Dados de audit com mais de 35 dias raramente são consultados de forma urgente. Quando necessários (inspeção), a latência de recuperação do S3 Glacier é aceitável.
🔍 Por que PITR no RDS?
DevOps
PITR (Point-in-Time Recovery) permite restaurar o banco para qualquer segundo dentro dos 35 dias — não apenas para o backup diário das 2h. Em ambiente GxP, se houver um incidente técnico que corrompa dados, você precisa demonstrar que os registros de audit anteriores ao incidente estão íntegros. Um backup diário deixa uma janela de até 24h sem cobertura. PITR fecha essa janela.
🔒 Por que Object Lock no S3?
DevOps · Qualidade
Object Lock no modo Compliance significa que o objeto não pode ser deletado ou sobrescrito por nenhuma entidade — nem o root account AWS, nem via API, nem por ordem judicial antes do período expirar. Isso transforma "prometemos que não alteramos" em uma garantia técnica demonstrável para o inspetor. FDA 21 CFR Part 11 §11.10(e) exige proteção contra alteração — Object Lock WORM é a implementação mais direta em cloud.
05 Controle de Acesso — Quem pode fazer o quê Imutabilidade por banco · Não por policy · Princípio do menor privilégio
🔵 ebr_app_role Aplicação
ebr_core — SELECT / INSERT / UPDATE / DELETE
ebr_audit.audit_events — INSERT / SELECT
✗ UPDATE em audit_events — REVOKED
✗ DELETE em audit_events — REVOKED
🟡 ebr_readonly_role Qualidade / Auditoria
ebr_audit.audit_events — SELECT only
ebr_core — SELECT only (views controladas)
✗ Sem acesso a INSERT / UPDATE / DELETE
✗ Sem acesso a tabelas de sistema
🔴 ebr_dba_role DBA · Processo controlado
ebr_core — acesso completo (migrations)
✗ ebr_audit — SELECT only via role separada
✗ Acesso direto ao ebr_audit bloqueado
⚠ Qualquer acesso requer aprovação de Qualidade
🔒
Regra não negociável: a role da aplicação (ebr_app_role) tem REVOKE UPDATE, DELETE na tabela audit_events em nível de banco. Isso significa que mesmo que um desenvolvedor escreva código com UPDATE na tabela de audit, o PostgreSQL rejeita a operação. A imutabilidade é garantida pelo banco — não por code review.
06 Boas Práticas Aplicadas — Referências de mercado O que cada decisão segue · Para o time de Arquitetura e DevOps
📐 Separation of Concerns no Audit
Arquitetura
Schema ebr_audit separado do ebr_core na mesma instância RDS. Write de audit não concorre com write operacional no mesmo tablespace. Buffer pool independente. WAL independente. Padrão adotado em sistemas regulados (SAP S/4, Veeva Vault, Oracle EBS).
🔐 Principle of Least Privilege
DevOps · Segurança
Cada role tem exatamente as permissões que precisa — nem mais. A aplicação não pode apagar audit. A equipe de qualidade não pode alterar dados operacionais. O DBA não acessa audit trail diretamente. Padrão NIST 800-53 AC-6.
📦 Immutable Audit Log Pattern
Arquitetura · Qualidade
Registros de audit são append-only por contrato de banco (REVOKE) e por contrato de infraestrutura (S3 Object Lock). Padrão recomendado por OWASP para audit trails em sistemas críticos. Equivalente ao event sourcing para dados de compliance.
🏗️ Defense in Depth
Arquitetura
Interceptor + Trigger são duas camadas independentes. Se uma falha ou for bypassada, a outra ainda captura. Não é redundância — é defense in depth. Padrão de segurança em sistemas onde a cobertura do audit é requisito regulatório não negociável.
💾 Tiered Storage (Hot/Cold)
DevOps · FinOps
Dados quentes (35 dias) em RDS para acesso operacional. Dados frios (5 anos) em S3 Glacier para compliance. Padrão AWS Well-Architected Framework — Cost Optimization Pillar. Redução estimada de 80-90% no custo de armazenamento de longo prazo vs. manter tudo em RDS.
🔍 Observability no Audit Trail
DevOps
Alerta Datadog configurado para falha na escrita do audit_events — zero tolerância. Campo source (INTERCEPTOR vs TRIGGER) permite detectar bypasses do ORM em produção. Dashboard Grafana para área de Qualidade — volume de eventos por módulo e por site sem acesso à API técnica.
⚠️
Validação pendente — Consultoria Especializada em Compliance GxP (*): a abordagem de dois schemas na mesma instância PostgreSQL (ebr_core + ebr_audit) requer validação formal com consultoria especializada antes de ser considerada decisão final para fins regulatórios. Esta arquitetura representa a direção técnica proposta — não é declaração de conformidade com ANVISA RDC 658/2022, GMP Annex 11 ou FDA 21 CFR Part 11.
