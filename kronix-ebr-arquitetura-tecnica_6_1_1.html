02 Separação de Responsabilidade entre Comandos e Consultas (CQRS) Padrão da camada Application · Commands → PostgreSQL · Queries → Elasticsearch
📖
O que é
Conceito base do padrão
Toda operação no sistema pertence a uma de duas categorias: ela muda alguma coisa ou ela apenas lê. Um Command representa uma intenção de mudança — criar um lote, assinar uma etapa, aprovar um dossiê. Uma Query busca informação sem alterar nada — listar OPs em execução, consultar o histórico de um equipamento. O CQRS formaliza essa separação fazendo com que cada tipo siga um pipeline de código independente, com seu próprio fluxo de validação, persistência e resposta.
🎯
Por que adotamos no Kronix EBR
Motivações concretas do projeto
GxP
Audit Trail limpo e previsível. Todo Command que altera estado passa pelo EF Core Interceptor e gera um registro de auditoria. Queries nunca disparam auditoria acidentalmente — a separação é estrutural, não depende de disciplina individual do desenvolvedor.
DADOS
Escrita e leitura em fontes diferentes. Commands persistem no PostgreSQL (ebr_core) — fonte da verdade. Queries de consulta e relatório leem do Elasticsearch — otimizado para busca e filtros complexos sem concorrer com as escritas transacionais.
MANUTENÇÃO
Rastreabilidade de intenção. Um Command tem nome explícito — SignStepCommand, ApproveDossierCommand. Fica claro o que o sistema permite fazer, quem pode fazer e quais validações se aplicam. Facilita a validação de sistemas (VSC) e a demonstração de controles em auditorias regulatórias.
⚙️
Como se aplica no projeto — estrutura mínima
Sem framework externo · Padrão nativo C# · Camada Application
Command — muda estado 
// Representa a intenção de assinar um step
public record SignStepCommand(
    Guid BatchRecordId,
    Guid StepId,
    string ActorId,
    string Reason
);

// Executa a intenção com validação e persistência
public class SignStepHandler
{
    private readonly IEbrDbContext _db;

    public SignStepHandler(IEbrDbContext db)
        => _db = db;

    public async Task Handle(
        SignStepCommand cmd,
        CancellationToken ct)
    {
        var step = await _db.Steps
            .FindAsync(cmd.StepId, ct);

        step.Sign(cmd.ActorId, cmd.Reason);
        // EF Interceptor captura a mudança
        // e grava no ebr_audit automaticamente

        await _db.SaveChangesAsync(ct);
    }
}
Query — apenas lê 
// Representa o que se quer consultar
public record GetBatchSummaryQuery(
    Guid BatchRecordId
);

// O que a query retorna (projeção)
public record BatchSummaryDto(
    Guid Id,
    string Status,
    string RecipeName,
    DateTimeOffset StartedAt
);

// Lê do Elasticsearch — sem tocar no PostgreSQL
public class GetBatchSummaryHandler
{
    private readonly ISearchClient _search;

    public GetBatchSummaryHandler(
        ISearchClient search)
        => _search = search;

    public async Task<BatchSummaryDto> Handle(
        GetBatchSummaryQuery qry,
        CancellationToken ct)
    {
        return await _search
            .GetBatchSummaryAsync(qry.BatchRecordId, ct);
    }
}
Fluxo de uma requisição
▶ Command (escrita)
Controller recebe request
→ instancia SignStepCommand
→ SignStepHandler.Handle()
→ valida regras de negócio GxP
→ persiste no PostgreSQL (ebr_core)
→ EF Interceptor grava em ebr_audit
→ retorna confirmação (sem payload)
▶ Query (leitura)
Controller recebe request
→ instancia GetBatchSummaryQuery
→ GetBatchSummaryHandler.Handle()
→ lê do Elasticsearch
→ sem tocar em ebr_core
→ sem disparar Audit Trail
→ retorna DTO projetado
ℹ️
Sem MediatR ou framework externo. O despachante de Commands e Queries é implementado diretamente via injeção de dependência do ASP.NET Core — sem overhead de biblioteca de terceiros. Cada Handler é registrado no container e chamado pelo Controller ou pelo Consumer SQS/Kafka correspondente.

04 Camada de Dados — Amazon RDS PostgreSQL + Amazon S3 Multi-AZ · Encryption at rest (AWS KMS) · Retenção regulatória 5 anos
🐘 Amazon RDS PostgreSQL — Por site · Dois schemas por instância · Backup 35 dias · Multi-AZ
📋 Schema: ebr_core — Aplicação SELECT / INSERT / UPDATE / DELETE
Dados operacionais do EBR. Role da aplicação com permissões completas nas tabelas de negócio.
Tabelas principais

🔒 Schema: ebr_audit — Audit Trail Somente INSERT · REVOKE UPDATE/DELETE
Schema dedicado. Role da aplicação tem apenas permissão de INSERT — UPDATE e DELETE revogados via GRANT no banco.

🪣 Amazon S3 — Armazenamento de Longo Prazo
S3 GLACIER + OBJECT LOCK (WORM) — Audit Trail
Destino: exportação diária dos registros do schema ebr_audit
Retenção: mínimo 5 anos — imutável após gravação (Write Once Read Many)
Object Lock: impede deleção ou sobrescrita mesmo por administrador AWS
Criptografia: SSE-KMS com chave dedicada ao EBR
S3 STANDARD — Anexos Operacionais
Destino: fotos de processo, laudos não regulatórios, anexos de desvio
Acesso: via pre-signed URL com expiração — nunca exposto publicamente
Retenção: política por tipo de documento — a confirmar com área de Qualidade
RDS Backup: snapshot automático diário · retenção 35 dias · restauração point-in-time disponível (PITR)

Definição arquitetural — Módulo vs. Feature: Um módulo tem modelo de dados próprio, ciclo de vida independente e pode ser acessado fora do contexto de outro módulo. Uma feature (◈) só existe no contexto de um módulo pai — não possui entidade raiz própria nem rota de acesso autônoma.
